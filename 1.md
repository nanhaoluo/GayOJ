## 一、项目概述

### 1.1 项目背景

ACM/ICPC 竞赛及算法训练需求持续增长，现有开源 OJ 系统在以下方面存在不足：

- **多赛制兼容性差**：多数系统仅支持单一赛制，无法同时满足 ACM、OI、IOI 等需求
- **题型单一**：大多仅支持代码题，缺少填空题、选择题等多样化题型支持
- **多角色协作缺失**：缺少教练端、裁判端等角色支持，无法满足教学管理场景
- **客观题离线训练能力弱**：多数系统缺少填空题、选择题的断网练习与本地即时判分能力
- **题库管理不便**：批量导入导出、试卷生成等功能不完善

本项目旨在构建一个覆盖**「选手训练 — 教练管理 — 裁判执法 — 系统运维」**全链路的综合算法训练平台，支持**代码题、填空题、选择题**等多种题型。

### 1.2 项目目标

| 目标 | 说明 |
| :--- | :--- |
| **多赛制支持** | 原生支持 ACM、OI/OJ、IOI、Codeforces 等主流赛制，提供自定义赛制规则引擎 |
| **多题型支持** | 支持代码题、填空题、选择题（单选/多选）等多种题型，满足刷题、作业、测验和比赛场景 |
| **在线代码评测** | 代码题通过在线沙箱评测，不在本地 CLI 中执行或评判代码 |
| **客观题离线训练** | 填空题、选择题可通过本地 CLI 离线作答与评判，适配断网训练和课堂练习 |
| **教学一体化** | 教练端支持训练计划制定、作业布置、学生能力分析，刷题与竞赛训练无缝融合 |
| **高扩展性** | 代码评测节点可水平扩展，客观题评判服务无状态扩容，容器化部署，模块化单体起步，未来可拆分微服务 |

### 1.3 目标用户

| 角色             | 典型用户                      | 核心需求                           |
| :--------------- | :---------------------------- | :--------------------------------- |
| **选手**   | ACM/OI 参赛者、算法爱好者     | 刷题训练、参加比赛、查看成绩与排名 |
| **教练**   | 高校/培训机构教师、集训队教练 | 管理学生、布置训练、分析学生能力   |
| **裁判**   | 比赛出题人、现场裁判          | 监控比赛、处理申诉、手动判题       |
| **管理员** | 系统运维人员、平台管理者      | 系统配置、用户管理、运维监控       |

### 1.4 项目范围

**In Scope：**

- 在线代码评测引擎（提交队列、沙箱执行、结果回写）
- 客观题即时评判（填空题、单选题、多选题）
- 客观题离线训练工具（仅填空题、选择题，不包含代码离线评判）
- 题库管理（CRUD、导入导出、题单试卷）
- 比赛系统（多赛制、实时排名、Clarification）
- 多角色端（选手/教练/裁判/管理员）
- 数据统计与分析
- API 开放平台
- 轻量讨论与题解功能
- 社交网络完整版

**Out of Scope（v1.0 不含）：**

- 代码题在 CLI 端执行或本地判题能力
- AI 智能出题 / 推荐
- 移动端原生 APP（仅 Web 响应式）
- 支付 / 商业化系统

## 二、系统总体架构

### 2.1 架构分层

```text
┌──────────────────────────────────────────────────────────────┐
│ Presentation Layer（展示层）                                  │
│ Web 前端 · Mobile Web · WebSocket Client · 客观题离线 CLI     │
└──────────────────────────────┬───────────────────────────────┘
                               │
┌──────────────────────────────▼───────────────────────────────┐
│ Gateway Layer（网关层）                                        │
│ Nginx / Caddy · API Gateway · JWT 认证 · 限流 · WAF           │
└──────────────────────────────┬───────────────────────────────┘
                               │
┌──────────────────────────────▼───────────────────────────────┐
│ Business Service Layer（业务层）                              │
│ 用户服务 · 题目服务 · 提交服务 · 比赛服务 · 权限服务           │
│ 客观题评判服务 · 训练服务 · 通知服务 · 统计服务               │
└──────────────────────────────┬───────────────────────────────┘
                               │
┌──────────────────────────────▼───────────────────────────────┐
│ Online Code Judge Layer（在线代码评测层）                      │
│ 评测队列 · 调度器 · 沙箱节点 · SPJ/交互器 · 结果收集器         │
│ 注：仅处理代码题；填空题/选择题不进入沙箱评测队列              │
└──────────────────────────────┬───────────────────────────────┘
                               │
┌──────────────────────────────▼───────────────────────────────┐
│ Data Layer（数据层）                                           │
│ PostgreSQL · Redis · MinIO/OSS · Elasticsearch · ClickHouse   │
└──────────────────────────────────────────────────────────────┘
```

> 客观题离线 CLI 只下载题面、选项、填空规则和经授权签名的判题规则包，在本地完成填空题/选择题训练评判；代码题始终通过在线代码评测层完成。

### 2.2 技术选型

| 层级 | 技术 | 说明 |
| :--- | :--- | :--- |
| 前端框架 | Vue 3 + TypeScript | SPA 单页应用，Vite 构建 |
| 代码编辑器 | Monaco Editor | 代码题在线编辑，支持语法高亮、自动补全 |
| 客观题组件 | Vue 表单组件 + 本地状态管理 | 填空题、单选题、多选题作答与结果展示 |
| 后端框架 | Go (Gin/Fiber) 或 Python (FastAPI) | Go 高并发首选；Python 快速开发备选 |
| 主数据库 | PostgreSQL 16+ | JSONB 支持多题型配置、全文搜索能力 |
| 缓存 / 消息队列 | Redis 7+ / Kafka | 在线代码评测队列、排行榜缓存、Session 管理 |
| 对象存储 | MinIO / 阿里云 OSS | 代码题测试数据、题面附件、离线训练包存储 |
| 搜索引擎 | Elasticsearch 8+ | 题目搜索、讨论搜索 |
| 分析引擎 | ClickHouse | 大规模提交日志分析、训练统计报表 |
| 代码沙箱 | Docker + nsjail / gVisor | 仅用于代码题在线隔离执行 |
| 客观题评判器 | 后端规则引擎 + CLI 本地规则引擎 | 填空题/选择题即时评判和离线评判，规则保持一致 |
| 容器编排 | Docker Compose / Kubernetes | 开发用 Compose，生产用 K8s |
| CI/CD | GitHub Actions / GitLab CI | 自动化测试、构建、部署流水线 |
| 监控 | Prometheus + Grafana | API、代码评测节点、客观题评判指标可视化 |
| 日志 | ELK Stack / Loki | 集中日志收集、查询与告警 |

## 三、用户角色与权限体系

### 3.1 RBAC 权限模型

采用 **RBAC（Role-Based Access Control）** 模型，支持自定义角色与细粒度权限控制。

```

User  ──M:N──▶  Role  ──M:N──▶  Permission
Role  ──M:N──▶  Resource    # 角色可绑定资源范围

```

**权限粒度示例：**

| 权限代码 | 说明 |
| :--- | :--- |
| `problem:create` | 创建题目 |
| `problem:edit:own` | 编辑自己创建的题目 |
| `problem:edit:all` | 编辑所有题目 |
| `contest:manage` | 创建和管理比赛 |
| `submission:override` | 手动覆盖评测结果 |
| `user:ban` | 封禁用户 |
| `system:config` | 修改系统配置 |

### 3.2 角色功能矩阵

| 功能 | 访客 | 选手 | 教练 | 裁判 | 管理员 |

| 浏览公开题目 | ✓ | ✓ | ✓ | ✓ | ✓ |
| 提交代码/答案 | — | ✓ | ✓ | ✓ | ✓ |
| 客观题离线训练 | — | ✓ | ✓ | ✓ | ✓ |
| 参加公开比赛 | — | ✓ | ✓ | ✓ | ✓ |
| 创建/管理题目 | — | — | ✓ | ✓ | ✓ |
| 创建/管理比赛 | — | — | ✓ | ✓ | ✓ |
| 管理班级/队伍 | — | — | ✓ | — | ✓ |
| 监控比赛/手动判题 | — | — | — | ✓ | ✓ |
| 处理 Clarification | — | — | — | ✓ | ✓ |
| 用户管理/封禁 | — | — | — | — | ✓ |
| 系统配置 | — | — | — | — | ✓ |

### 3.3 选手端功能

- 个人主页与竞赛档案（AC 记录、参赛历史、Rating 曲线）
- 代码提交历史与管理（查看、对比、收藏自己的提交）
- 错题本 / 收藏夹 / 做题计划
- 比赛报名与赛时提交
- 讨论区 / 题解发布（评论、发布题解）
- 通知中心（评测结果推送、比赛提醒）
- 排行榜（全局 / 周 / 月 / 学校 / 班级）

### 3.4 教练端功能

- 队伍/班级管理（邀请码创建、批量导入学生）
- 训练计划制定与跟踪
- 学生能力分析面板（知识点掌握度、薄弱项雷达图）
- 布置作业/训练赛（设定截止时间）
- 查看学生提交代码（防抄袭辅助）
- 自定义题单/试卷发布

### 3.5 裁判端功能

- 比赛实时监控面板（实时提交流、异常检测）
- Clarification 审批与广播
- 手动判题 / Override（覆盖自动评测结果）
- 气球管理（ACM 赛制物理气球发放跟踪）
- 作弊检测辅助（代码相似度对比、提交时间模式分析、IP/设备指纹）

### 3.6 管理员端功能

- 用户管理（角色分配、封禁、批量操作）
- 代码评测节点管理（状态监控、上下线、优先级配置）
- 系统全局配置（参数管理、公告管理、首页推荐）
- 操作日志与审计（操作日志、异常提交日志）
- 数据备份与恢复

## 四、功能模块设计

系统整体功能模块划分如下：

```text
OJ System
├── 用户系统
│   ├── 注册 / 登录（用户名密码 / OAuth2 / SSO）
│   ├── 个人资料管理
│   ├── 角色与权限分配
│   ├── 第三方认证对接
│   └── 账号安全（密码加密 / 2FA / 登录日志）
│
├── 评测与训练
│   ├── 代码题在线评测（Web 提交 → 队列 → 沙箱执行）
│   ├── 客观题即时评判（填空题 / 单选题 / 多选题）
│   ├── 客观题离线训练（CLI 本地答题与判分，不执行代码）
│   ├── 编译器管理（多语言 / 多版本）
│   ├── Special Judge / 交互题 / 通信题（仅代码题）
│   ├── 评判结果管理（代码题状态码 / 客观题逐项判分）
│   ├── 代码评测节点调度（负载均衡 / 优先级）
│   └── 代码沙箱安全（Docker / nsjail / syscall 白名单）
│
├── 题库系统
│   ├── 题目 CRUD（Markdown / LaTeX 题面）
│   ├── 代码题测试数据管理（上传 / 下载 / 校验）
│   ├── 客观题公开配置与判题配置管理
│   ├── 题目标签与分类（多级知识点体系）
│   ├── 难度评估体系
│   ├── 题目版本控制
│   ├── 多语言题面（i18n）
│   ├── 题单 / 试卷管理（公开 / 私密 / 考试模式）
│   └── 批量导入 / 导出（FPS / QDU / Hydro 等格式）
│
├── 比赛系统
│   ├── ACM 赛制（封榜 / 罚时 / First Blood / 气球）
│   ├── OI 赛制（子任务评分 / 多次提交取最高）
│   ├── IOI / CF 赛制（动态分值 / Hack）
│   ├── 自定义赛制规则引擎
│   ├── 团队赛制
│   ├── 虚拟参赛（Virtual Contest）
│   ├── 赛时实时排行榜（WebSocket）
│   ├── Clarification 系统
│   ├── 防作弊（代码相似度 / 行为分析）
│   └── 赛后重测
│
├── 社交系统
│   ├── 讨论板块（按题目 / 比赛分区）
│   ├── 题解系统（Markdown + 代码高亮）
│   ├── 评论与回复
│   ├── 关注 / 粉丝机制
```

## 五、评测与训练详细设计

本章将代码题在线评测、填空题/选择题即时评判、客观题离线训练拆分设计，避免将代码题离线执行纳入系统边界。

### 5.1 评测与训练流程

**代码题评测流程：**

```
用户提交 → 参数校验 → 写入数据库(PENDING) → 推入评测队列
    → 调度器分配评测节点 → 创建沙箱 → 编译代码 → 逐测试点执行
    → 对比输出 → 汇总结果 → 写回数据库 → WebSocket 推送用户
```

**填空题/选择题评测流程：**

```
用户提交 → 参数校验 → 写入数据库(PENDING) → 即时评判（无需队列）
    → 答案对比（字符串匹配/选项匹配） → 计算得分 → 写回数据库 → WebSocket 推送用户
```

**详细步骤（代码题）：**

1. **提交校验**：验证语言合法性、代码长度限制、提交频率限制、比赛状态检查
2. **入队**：写入数据库 `status=PENDING`，推入 Redis/Kafka 队列
3. **调度**：调度器根据节点负载、支持语言、优先级（比赛 > 刷题）分配任务
4. **编译**：在沙箱内编译代码，捕获编译错误，限制编译时间和资源
5. **执行**：逐测试点运行，限制时间/内存/进程数/输出大小
6. **对比**：Exact Match / Special Judge / 交互协议判定
7. **汇总**：合并所有测试点结果，计算总分（OI 模式）、总用时/内存
8. **回写**：更新数据库状态，WebSocket 推送用户

**详细步骤（填空题/选择题）：**

1. **提交校验**：验证答案格式、提交频率限制、比赛状态检查
2. **即时评判**：
   - **填空题**：逐空对比答案，支持大小写敏感/不敏感、去除首尾空格、正则匹配
   - **单选题**：对比选项ID，完全匹配得分
   - **多选题**：对比选项ID集合，完全匹配得分（部分系统支持部分分）
3. **计算得分**：累加各空/题得分
4. **回写**：直接写入数据库 `status=AC/WA`，返回结果给用户

### 5.2 评测状态码

| 状态码 | 英文 | 说明 |
| :--- | :--- | :--- |
| `PENDING` | Pending | 等待评测 |
| `JUDGING` | Judging | 评测中 |
| `AC` | Accepted | 通过 |
| `WA` | Wrong Answer | 答案错误 |
| `TLE` | Time Limit Exceeded | 超时 |
| `MLE` | Memory Limit Exceeded | 超内存 |
| `RE` | Runtime Error | 运行时错误 |
| `OLE` | Output Limit Exceeded | 输出超限 |
| `CE` | Compile Error | 编译错误 |
| `PE` | Presentation Error | 格式错误 |
| `SE` | System Error | 系统内部错误 |
| `ILE` | Idleness Limit Exceeded | 空闲超时（交互题） |

### 5.3 代码沙箱安全机制

**资源限制：**

| 资源 | 范围 | 说明 |
| :--- | :--- | :--- |
| 时间限制 | 0.5s ~ 10s | 逐测试点计时 |
| 内存限制 | 64MB ~ 1024MB | 含栈空间 |
| 输出限制 | 通常 64MB | 超出即 OLE |
| 进程数限制 | 配置化 | 防止 fork 炸弹 |
| 文件大小限制 | 配置化 | 防止磁盘写满 |

**隔离策略：**

- 容器级隔离（Docker / gVisor）
- 系统调用白名单（seccomp filter）
- 网络访问完全禁止
- 文件系统只读挂载（仅测试数据目录可读）
- 临时目录隔离，评测结束即销毁
- UID/GID 隔离，无 root 权限

### 5.4 代码题特殊评测模式

以下模式仅适用于代码题或传统算法题，不用于填空题/选择题离线训练。

| 模式 | 说明 | 实现方式 |
| :--- | :--- | :--- |
| **标准评测** | Exact Match，去除末尾空白 | diff 对比 |
| **Special Judge** | 自定义判定逻辑（多解 / 浮点精度） | 题目提供 checker 程序 |
| **交互题** | 选手程序与交互器通信 | 管道（pipe）通信 |
| **通信题** | 选手实现指定接口，评测调用 | 动态库链接 |
| **提交答案题** | 上传传统算法题输出文件，由在线系统比对 | 无需编译执行，但仍属于在线评测 |
| **子任务评分** | OI 模式按测试点组给分 | Subtask 配置，部分分累加 |

### 5.5 代码评测节点调度

- 代码评测节点通过**心跳机制**注册到调度器
- 支持**动态上下线**，不影响正在运行的代码评测任务
- 填空题/选择题由业务层客观题评判服务即时处理，不进入代码评测队列
- 调度器根据以下因素分配代码题任务：
  - 节点当前负载（CPU / 内存 / 队列长度）
  - 节点支持的编程语言
  - 任务优先级（比赛中提交 > 日常刷题）
- 生产环境建议代码评测节点**独立于业务服务器**部署
- 支持**横向扩展**：添加节点即可提升代码评测吞吐

### 5.6 本地训练工具

提供 CLI 工具 `oj-cli`，支持**填空题、选择题**离线训练。本工具不编译、不运行、不评判代码题；代码题统一通过在线代码评测层处理。

**功能：**

- 同步题单、试卷、题面、选项、填空规则与离线训练包
- 支持填空题本地答题、即时判分与结果缓存
- 支持单选题/多选题本地答题、即时判分与结果缓存
- 支持答案文件导入、交互式答题、批量客观题判分
- 支持训练结果在恢复联网后回传在线系统
- 支持 Windows / macOS / Linux

**使用示例：**

```bash
# 登录与同步
oj-cli login --server https://oj.example.com
oj-cli pull-set 9001 --offline         # 拉取题单/试卷离线训练包

# 填空题训练
oj-cli answer 2001 --interactive       # 交互式答题
oj-cli answer 2001 --file answers.txt  # 从文件读取答案并本地判分

# 选择题训练
oj-cli answer 3001 --interactive       # 交互式答题
oj-cli answer 3001 --file answers.json # 从文件读取答案并本地判分

# 结果同步
oj-cli sync-results                    # 恢复联网后同步离线训练结果
```

**答案文件格式：**

填空题答案文件（`answers.txt`）：
```
blank_1: 答案1
blank_2: 答案2
blank_3: 答案3
```

选择题答案文件（`answers.json`）：
```json
{
  "single_choice": "A",
  "multiple_choice": ["A", "C", "D"]
}
```

## 六、赛制与比赛管理

### 6.1 赛制对比

| 维度                  | ACM/ICPC                 | OI                 | IOI / Codeforces           |
| :-------------------- | :----------------------- | :----------------- | :------------------------- |
| **计分**        | 通过数量 + 罚时          | 总分（每题满分制） | 动态分值（越早 AC 越高分） |
| **提交限制**    | 无限制，每次错误累加罚时 | 多次提交取最高分   | 无限制                     |
| **封榜**        | 比赛结束前 1 小时冻结    | 通常不封榜         | 可选封榜                   |
| **部分分**      | 不支持（AC / WA 二元）   | 支持（子任务评分） | 不支持                     |
| **Hack**        | 不支持                   | 不支持             | 支持（找反例）             |
| **First Blood** | ✓ 标记首次 AC           | 可选               | 可选                       |
| **气球**        | ✓ 物理气球发放          | —                 | —                         |

### 6.2 比赛生命周期

```
创建比赛
    → 配置题目（分值 / 时限 / 数据）
    → 邀请 / 开放报名
    → 定时开赛
    → 赛中监控（提交 / Clarification / 公告）
    → 封榜（ACM 赛制，结束前 1 小时）
    → 比赛结束
    → 赛后重测（Hacking）
    → 终榜发布
    → 赛后整理（题解发布 / 讨论区开放 / 代码公开）
```

### 6.3 比赛管理功能

**赛前管理：**

- 比赛信息配置（名称、时间、赛制类型、描述、封面图）
- 题目选择与配置（分值、题型、代码题时限/测试数据、客观题判分规则、Special Judge）
- 报名管理（公开报名 / 密码 / 邀请码 / 白名单 / 最大人数限制）
- 比赛克隆（复制历史比赛配置，快速创建新比赛）

**赛中管理：**

- 实时排行榜（WebSocket 推送，支持排序/筛选）
- 提交监控面板（实时提交流，异常检测）
- Clarification 审批与广播（裁判回答后可选择广播给所有选手）
- 比赛暂停 / 延长 / 个别补时
- 异常提交检测告警（高频提交、相同 IP 等）

**赛后管理：**

- 统一重测（Hacking），验证评测结果一致性
- 终榜生成与发布
- 代码公开（可配置公开时间，如赛后 24 小时）
- 比赛统计报告自动生成
- Rating 更新（基于比赛结果计算选手 Rating 变化）

| `Hydro OJ`   | Hydro       | YAML + 文件包                       |
| `DomJudge`   | DomJudge    | problem.yaml + data/ 目录           |
| `LOJ`        | LibreOJ     | YAML + 测试数据目录                 |
| `ZIP 自定义` | 自定义规范  | 题面 Markdown + 测试数据 + SPJ 源码 |

**导入流程：** 上传文件 → 解析 → 预览确认 → 批量入库

**导出流程：** 选择题目 → 选择格式 → 打包下载

## 八、数据库设计

### 8.1 ER 关系总览

```
用户域:
  User  ──1:N──▶  Submission
  User  ──M:N──▶  Role
  User  ──M:N──▶  Team
  User  ──M:N──▶  Contest  (ContestParticipant)

内容域:
  Problem ──M:N──▶  Tag
  Problem ──M:N──▶  Contest  (ContestProblem)
  Problem ──M:N──▶  ProblemSet  (ProblemSetItem)
  Problem ──1:N──▶  Submission

比赛域:
  Contest ──1:N──▶  ContestProblem
  Contest ──1:N──▶  ContestParticipant
  Contest ──1:N──▶  ContestClarification
  Contest ──1:N──▶  Submission

团队域:
  Team  ──1:N──▶  TeamMember  ──N:1──▶  User

社交域:
  Discussion ──自关联──▶  Discussion  (回复关系)
```

## 十一、非功能性需求

### 11.1 性能要求

| 指标 | 目标值 | 说明 |
| :--- | :--- | :--- |
| API 响应时间 | < 200ms (P95) | 非评测类接口 |
| 代码评测排队延迟 | < 5s（正常负载） | 从代码提交到开始沙箱评测 |
| 客观题即时评判延迟 | < 300ms (P95) | 填空题/选择题在线提交后的判分延迟 |
| 单个代码评测节点吞吐 | ≥ 100 提交/分钟 | 含编译和执行 |
| 客观题离线包生成耗时 | < 10s / 1000 题 | 仅包含授权公开题面和判分规则 |
| 并发用户数 | ≥ 5000 | 比赛期间同时在线 |
| WebSocket 推送延迟 | < 500ms | 排行榜 / 通知 |
| 页面首屏加载 | < 2s | 首屏有意义内容 |

### 11.2 安全要求

**应用安全：**

- JWT + HTTPS 全链路加密传输
- 密码使用 bcrypt 或 argon2 加密存储
- CSRF / XSS / SQL 注入防护
- 输入校验（前后端双重）与输出编码
- 敏感操作二次验证（2FA）
- 登录失败锁定策略

**评测与训练安全：**

- 代码题使用 Docker / nsjail / gVisor 沙箱多层隔离
- 系统调用白名单（seccomp filter）
- 网络访问完全禁止（评测容器无网络）
- 资源严格限制（时间 / 内存 / 进程数 / 输出大小）
- 代码评测节点与业务服务网络隔离
- 代码评测结束立即清理临时文件
- 客观题标准答案与判分规则严格权限控制，不通过普通题目详情接口返回
- 客观题离线训练包需签名、防篡改，可配置有效期和同步策略

**数据安全：**

- 代码题测试数据加密存储
- 客观题 `judge_config` 加密或字段级权限隔离
- 提交代码、客观题作答记录定期归档
- 数据库连接使用 SSL
- 敏感配置项不入库（环境变量 / Vault 管理）

### 11.3 可用性与容灾

- 系统可用性 ≥ 99.9%（全年停机 < 8.76 小时）
- 数据库主从复制 + 自动故障切换（Patroni / repmgr）
- 代码评测节点集群化，单节点故障不影响整体服务
- 客观题即时评判服务无状态化部署，可随 API 服务水平扩展
- 比赛期间降级策略（关闭讨论区、统计分析等非核心功能，保障提交与评判）
- 数据每日自动备份，保留 30 天
- 备份定期恢复验证

### 11.4 可维护性

- 代码覆盖率 ≥ 70%（代码评测、客观题评判、权限系统等核心模块 ≥ 85%）
- API 向后兼容，版本化管理（`/api/v1`、`/api/v2`）
- 配置热更新（评测参数、系统参数无需重启服务）
- 完善的日志分级（DEBUG/INFO/WARN/ERROR）与结构化输出
- Prometheus 指标 + Grafana Dashboard + AlertManager 告警
- CI/CD 自动化流水线（lint → test → build → deploy）

## 十四、风险分析

| 风险 | 等级 | 影响 | 应对策略 |
| :--- | :--- | :--- | :--- |
| 代码评测沙箱逃逸 | **高** | 服务器被入侵 | 多层隔离（Docker + nsjail + seccomp），定期安全审计，最小权限原则 |
| 客观题答案泄露 | **高** | 训练/比赛公平性受损 | 公开配置与判题配置隔离，`judge_config` 加密存储，严格权限审计 |
| 离线训练包被篡改 | **高** | 本地判分结果不可信 | 离线包签名、防篡改校验、有效期控制、同步时二次校验 |
| 比赛期间系统崩溃 | **高** | 比赛中断，平台信任丧失 | 冗余部署 + 降级策略 + 赛前压测 + 比赛期间技术值守 |
| 代码评测节点性能瓶颈 | 中 | 代码评测队列堆积，用户体验差 | 节点水平扩展 + 优先级队列 + 提交限流 |
| 代码题测试数据泄露 | 中 | 比赛公平性受损 | 数据加密存储 + 访问权限严格控制 + 操作审计日志 |
| 代码抄袭 | 中 | 比赛公平性受损 | MOSS 代码相似度检测 + 提交行为分析 + IP/设备指纹 |
| 高并发提交导致数据库压力 | 中 | 系统整体响应变慢 | 写入队列化 + 读写分离 + 热数据缓存 + ClickHouse 分担统计查询 |
| LaTeX 题面渲染兼容性 | 低 | 部分题面数学公式显示异常 | KaTeX 优先渲染 + MathJax 兜底 + 预渲染静态化 |
| 前端编辑器性能 | 低 | 大文件编辑卡顿 | Monaco Editor 按需加载 + Web Worker 语法检查 |
